# macOS builds ‚Äî CRM Desktop

This document describes how to build and distribute **CRM Desktop** on macOS (Intel **x64** and Apple Silicon **arm64**), and how Apple **Gatekeeper** and **notarization** affect users.

---

## Prerequisites

- **macOS** (required to produce signed/notarized builds and DMGs reliably)
- **Node.js** 20+
- **Xcode Command Line Tools** (`xcode-select --install`)
- **JDK 17** (only for `npm run build:backend` if you run Maven locally)

Linux/Windows cannot produce native macOS `.app` / `.dmg` bundles in a supported way‚Äîuse a Mac or CI that runs on macOS (e.g. GitHub Actions `macos-latest`).

---

## What gets bundled

| Asset | Purpose |
|-------|---------|
| `CRM Desktop.app` | Electron shell + React (`app.asar`) |
| `Contents/Resources/backend/crm-backend.jar` | Spring Boot API |
| `Contents/Resources/jre/‚Ä¶` | Temurin 17 JRE for the **same CPU arch** as the DMG (Intel vs Apple Silicon) |
| User SQLite DB | `~/Library/Application Support/crm-desktop/data/crm-desktop.db` |

The **beforePack** hook (`scripts/before-pack.cjs`) copies the correct JRE from:

- `resources/jre-cache/darwin-arm64` ‚Üí arm64 builds  
- `resources/jre-cache/darwin-x64` ‚Üí x64 builds  

Populate the cache with:

```bash
npm run download:jre:mac-cache
```

## Build commands

From `CRM_FRONTEND`:

```bash
npm install

# One-time per release: backend JAR + both macOS JREs + PNG/ICO/ICNS
npm run predist:mac

# Vite production build + electron-builder for macOS (Intel + Apple Silicon)
npm run dist:mac
```

Artifacts (example for version `1.0.0`):

| Output | Description |
|--------|-------------|
| `release/CRM-Desktop-1.0.0-arm64.dmg` | Apple Silicon installer |
| `release/CRM-Desktop-1.0.0-x64.dmg` | Intel Mac installer |
| `release/mac-arm64/CRM Desktop.app` | Unpacked arm64 app (from `dir` target) |
| `release/mac-x64/CRM Desktop.app` | Unpacked x64 app |

**Universal binary (optional, advanced):** Electron can emit a **universal** app that runs on both architectures in a single `.app`. That requires a separate electron-builder configuration and approximately doubles app size; the current setup ships **two separate DMGs** (simpler for updates and smaller per download).

## Gatekeeper (unsigned builds)

**Gatekeeper** is macOS security: by default it only runs apps from the App Store or signed by identified developers.

### Unsigned / ad-hoc signed apps

- First open may show **‚ÄúCRM Desktop cannot be opened because the developer cannot be verified.‚Äù**
- Users can: **System Settings ‚Üí Privacy & Security** ‚Üí **Open Anyway**, or right-click the app ‚Üí **Open** ‚Üí confirm.

This is normal for open-source or internal builds that are **not** Apple-notarized.

### Signed but not notarized

- A **Developer ID** signed app may still be blocked on first run until the user allows it, depending on macOS version and signing settings.

### Notarized + stapled (recommended for public releases)

For a smooth experience **without** scary dialogs:

1. Enroll in the **Apple Developer Program** ($99/year).
2. Create **Developer ID Application** certificate.
3. **Code sign** the `.app` (electron-builder can automate with env vars).
4. **Notarize** with Apple (upload to Apple notary service).
5. **Staple** the ticket to the `.dmg` / `.app`.

electron-builder supports this via environment variables (see [electron-builder code signing](https://www.electron.build/code-signing)).

**Minimum for wide distribution:** Developer ID sign + **notarize** + staple‚Äînot just sign.

## Hardened runtime & entitlements

`package.json` enables:

- `hardenedRuntime: true` ‚Äî required for notarization.
- Entitlements files: `build/entitlements.mac.plist` and `build/entitlements.mac.inherit.plist` ‚Äî JIT and executable memory allowances used by Chromium/Electron.

The bundled **Java** process is launched as a child of Electron; inheriting entitlements reduces spurious kills when notarized. For strict enterprise policies, review entitlements with your security team.

## Backend auto-start on macOS

The same **Electron** `backendManager` is used:

- Resolves Java at `Contents/Home/bin/java` under the bundled macOS JRE layout.
- Spawns: `java -jar ‚Ä¶/crm-backend.jar` with `CRM_USER_DATA` and `CRM_DB_PATH`.
- Stops the JVM with **SIGTERM** on quit (not `taskkill`).

No separate MySQL install; SQLite lives under Application Support.

## Troubleshooting

| Issue | Action |
|------|--------|
| ‚ÄúDamaged app‚Äù / quarantine | `xattr -cr "/Applications/CRM Desktop.app"` or download from trusted release URL; distribute via HTTPS |
| Wrong architecture DMG | Installers are per-arch; M-series Macs need **arm64** DMG |
| `before-pack` missing cache | Run `npm run download:jre:mac-cache` before `dist:mac` |
| Java not found in app | Confirm `resources/jre/Contents/Home/bin/java` exists inside `.app` |

---

## CI suggestion (GitHub Actions)

Use `runs-on: macos-latest`, cache `node_modules` and `~/.cache/electron`, run `npm run predist:mac` and `npm run dist:mac`, upload `release/*.dmg` as release assets. Add signing secrets only when ready for production.
